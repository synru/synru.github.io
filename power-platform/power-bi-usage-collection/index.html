Resolving 30-Day Usage Data Limitation in Power BI

Power BI provides valuable insights into report usage through its built-in usage metrics. However, there's a significant limitation: the usage data is only available for the last 30 days. This can be problematic for organizations that need to track usage trends over longer periods for capacity planning, user adoption analysis, or compliance reporting.

If you have admin access, then you can adjust the setting. However, if you like me where I only have access to the 30 days data, then it's frustrating.

In this blog post, I'll show you how to overcome this limitation by using Power Automate to regularly query the Power BI usage dataset and persist the data in a long-term storage solution.

The Problem: 30-Day Limitation

Power BI's usage metrics dashboard shows data for the most recent 30 days only. While this is sufficient for short-term monitoring, it falls short when you need:

  • Historical usage trends over months or years
  • Capacity planning based on usage patterns
  • User adoption metrics for quarterly or annual reports
  • Compliance and audit requirements

The Solution: Automated Data Collection

The solution involves creating an automated workflow that:

  1. Uses Power Automate to query the Power BI Admin API for usage data
  2. Processes and transforms the data
  3. Persists the data to a long-term storage solution (like Azure SQL Database, SharePoint, or Dataverse)

Step-by-Step Implementation

Step 1: Check Workspace ID and Semantic Model GUID

You can collect the workspace and dataset GUIDs from the semantic model URL. The URL path contains the workspace ID and dataset ID which is in format https://app.powerbi.com/onelake/details/{workspaceId}/dataset/{datasetId}.

Step 2: Set Up Power Automate Flow

Create a new automated cloud flow in Power Automate that runs on a schedule (e.g., daily or weekly).

Trigger: Recurrence

Set up a recurrence trigger to run the flow at regular intervals. For daily collection, set it to run once per day.

Subject to your repository, retrieve the last activity date the collection provides and put into variables {Year}, {Month}, {Date}

Action: Run a query against a dataset

Workspace : {Workspace ID}

Dataset : {Dataset ID}

Query Text: EVALUATE FILTER({Table}, {Date_Column}>DATE({Year}, {Month}, {Date}))

Persist the result into your respository

Step 3: Process the Data

Once you have the usage data, aggregate/transform each table individually. According to your objectives, decide what tables to collect, e.g. Views, Rank, Report...

  • Report views and interactions
  • User activity
  • Data refresh patterns
  • Performance metrics

You can choose to use Power Automate to perform the job but personally I don't like as it's not transformation friendly. I prefer to trigger transformation within the persistence layer below

Step 4: Persist the Data

Store the processed data in a persistent storage solution. Options include:

Azure SQL Database

Use the SQL Server connector to insert data into a database table.

INSERT INTO PowerBIUsage (Date, ReportId, UserId, ActionType, Duration)
VALUES (@Date, @ReportId, @UserId, @ActionType, @Duration)

SharePoint List

Create a SharePoint list and use the SharePoint connector to add items.

Azure Table Storage or Blob Storage

For large volumes of data, use Azure Storage services.

Step 5: Handle Data Retention and Cleanup

Implement logic to:

  • Avoid duplicate entries
  • Manage data retention policies
  • Archive old data if necessary

Considerations

Permissions

Ensure your Power Automate flow has the necessary permissions:

  • Execute query on Power BI semantic model
  • Access to the target storage system

API Limits

Be aware of Power BI API rate limits and plan your data collection accordingly.

Data Volume

Usage data can be substantial. Consider data compression and efficient storage strategies.

Conclusion

By implementing this automated solution, you can collect and store Power BI usage data beyond the 30-day limitation, enabling better long-term analysis and decision-making. The key is to set up a reliable, automated process that runs consistently without manual intervention.

This approach not only solves the data limitation problem but also provides a foundation for more advanced analytics on Power BI adoption and usage patterns within your organization.
